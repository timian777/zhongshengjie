### ❌ 假修复（1 项 - **必须立即重做**）

#### C3【新 Critical】M7 向量维度统一是**假修复**

**提交：** `ea98c7fbb` 声称"unify all write paths to BGE-M3 1024 dimensions"，实际只改了数字没改模型。

**证据：**

`modules/knowledge_base/sync_manager.py`:
```python
# 第 87 行 - 维度说 1024 ✅
VECTOR_SIZE = 1024  # BGE-M3 dense 向量维度

# 第 170-178 行 - 模型还是 MiniLM（384 维）❌
def _load_model(self):
    if self._model is None:
        from sentence_transformers import SentenceTransformer
        self._model = SentenceTransformer(
            "paraphrase-multilingual-MiniLM-L12-v2"  # ❌ 还是 384 维模型
        )
    return self._model

# 第 296/421/528 行 - encode 调用方式也是旧的
vectors = model.encode(texts, show_progress_bar=True, batch_size=32)
```

`.case-library/scripts/sync_to_qdrant.py` **同样的问题**（第 61 行 VECTOR_SIZE=1024，第 192 行模型还是 MiniLM）。

**后果（比修复前更严重）：**
- 修复前：全局一致 384 维，能跑通但与 BGE-M3 检索不兼容
- 修复后（当前）：Qdrant Collection 创建时 size=1024，但 model.encode() 产出 384 维向量
- **下次任何人触发同步 → 立即报 `Vector dimension mismatch: expected 1024, got 384` → 整个入库流程崩溃**

**修复难点说明给 GLM5：** 不是只改常数，要完整替换 3 处：
1. `from sentence_transformers import SentenceTransformer` → `from FlagEmbedding import BGEM3FlagModel`
2. 模型实例化参数和类名
3. `model.encode(texts, show_progress_bar=True, batch_size=32)` → BGE-M3 返回的是 `{"dense_vecs": np.ndarray}` 字典，需要取 `["dense_vecs"]`

## 3. 新发现的问题

### C4【新 Critical】pause_workflow 意图落入路由死角

**位置：** `core/conversation/intent_classifier.py:92`

```python
"pause_workflow": {
    "patterns": [r"暂停", r"等一下", r"先停", r"休息一下"],
    "category": IntentCategory.WORKFLOW,  # ❌ 应该是 WORKFLOW_CONTROL
    "entities": [],
},
```

**路由流程：**

```
用户："暂停"
  ↓
IntentClassifier.classify() → category=WORKFLOW
  ↓
_handle_special_intents() → 未匹配（只处理 undo_last/query_progress/continue_workflow/view_history）
  ↓
_execute_intent() → category==WORKFLOW ≠ WORKFLOW_CONTROL → 落到 else 分支
  ↓
IntentRouter.route("pause_workflow") → _get_handler("pause_workflow") → None
  ↓
返回 RoutingResult(success=False) → 用户看到"意图未处理"
```

**实际已写好但不可触达的代码：** `conversation_entry_layer.py:482-510` 的 pause_workflow 处理逻辑在 `_execute_workflow_control` 里，但因 category 不匹配永远执行不到。

**修复：** `intent_classifier.py:92` 把 `IntentCategory.WORKFLOW` 改为 `IntentCategory.WORKFLOW_CONTROL`。

**注：** `continue_workflow`、`undo_last` 的 category 也是 WORKFLOW，但它们在 `_handle_special_intents` 中按 intent 名字匹配处理，不依赖 category，所以不受此问题影响。

### 5.2 代码分层与编译策略

项目代码按**编译可行性 × 保护价值**分四类：

#### 类 A — 核心算法，**必须编译**

高算法密度、字符串占比低、是真正的 IP：

| 目录/文件 | 行数 | 保护理由 |
|----------|------|---------|
| `core/inspiration/` 整目录 | ~3000 | 灵感引擎核心（鉴赏师/变体生成器/结构分析器）|
| `core/retrieval/unified_retrieval_api.py` | 1049 | 多 Collection 检索路由逻辑 |
| `core/conversation/file_updater.py` | ~800 | 文件增量更新算法 |
| `modules/knowledge_base/sync_manager.py` | ~700 | 向量同步管道 |
| `.novel-extractor/extractors/` | ~7000 | 11 类提取算法（技法/情感/伏笔/角色关系/世界观）|
| `.novel-extractor/scorers/quality_scorer.py` | ~600 | 质量打分算法 |

#### 类 B — 算法+大量中文字符串，**编译前需重构**

Cython `.pyd` 用 `strings` 命令就能提取所有字符串常量，所以意图匹配表这种**规则即字符串**的代码，编译的保护意义很低。

| 文件 | 行数 | 阻塞原因 | 预处理方案 |
|------|------|---------|-----------|
| `core/conversation/intent_classifier.py` | 763 | 459 个中文正则/映射硬编码 | **字符串外化到 `config/intent_patterns.json`**，代码只保留加载和匹配逻辑 |
| `.vectorstore/core/workflow.py` | 2721 | 大量中文文档和业务文案 | 合并到新框架时顺便拆分 |
| `core/conversation/missing_info_detector.py` | ~500 | 字段名和提示语中文 | 抽到 `config/missing_info_rules.json` |

**"字符串外化"的含义：**

```python
# 改造前（编译保护效果低）
patterns = {
    r"加个角色(.+?)": IntentCategory.CHARACTER,
    r"设置势力(.+?)": IntentCategory.FACTION,
    # ... 459 条
}

# 改造后（算法在 .pyd，配置在 JSON）
patterns = load_patterns_from_json("config/intent_patterns.json")
# .pyd 里只有加载/匹配算法，无规则字符串
```

JSON 本身是明文，但那是**业务规则**，不是 IP；真正的 IP 是匹配 + 分类 + 路由决策算法。
